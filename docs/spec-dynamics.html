Dynamic Spec Schema

Dynamic specs define the runtime behavior that emerges from mechanics interacting with player input and each other over time. They answer: "What BEHAVIOR should emerge during play?"

File pattern: specs/dynamics/{feature}.dyn.md
ID format: DYN-{NNN}
Scaffold: npx mda new dynamic "Behavior Name"

From the MDA paper

"Dynamics describes the run-time behavior of the mechanics acting on player inputs and each others' outputs over time."

Template

Frontmatter

---
id: DYN-{NNN}
name: {Descriptive name}
traces_to_aesthetics: [AES-NNN]
---

Required Sections

Behavior Description

2-3 sentences describing the emergent behavior. Focus on what the PLAYER observes and what the SYSTEM produces — not how it's coded.

Feedback System

Model the dynamic as feedback loops. Each loop is classified as positive (amplifying) or negative (dampening).

### Loop: Discovery Pacing
- Type: Negative
- Cycle: Player searches → time increases → hints escalate → discovery becomes easier
- Effect: Prevents frustration, ensures Discovery aesthetic is delivered

Include an ASCII diagram showing player actions, system states, and feedback arrows with +/- labels.

Interaction Patterns

Key interactions between player and system. More specific than the feedback model.

### Pattern: Hint Escalation
- Trigger: Timer passes threshold without discovery
- Sequence: Audio cue → visual glow → particle trail → beacon
- Duration: 30 seconds (escalates in 4 tiers)
- Outcome: Player guided to target

Invariants

Conditions that MUST hold. These are hard constraints that code validates against.

- INV-1: Discovery MUST occur within 50s of search start
- INV-2: Every reveal MUST have at least 1s of partial visibility before full reveal
Invariants are non-negotiable

Code that violates an invariant is WRONG even if the game "feels okay" in testing. Invariants are validated at runtime by the MDA Logger.

Degenerate Dynamics

Behaviors that could emerge but undermine the target aesthetics. Each must have a detection method and prevention strategy.

- Camping: Player stands still and waits for hints to solve everything
  - Breaks: Discovery, Challenge
  - Detection: Player velocity = 0 for >10s while hints are active
  - Prevention: Hints only activate in new areas, not repeated locations

Rules

  1. Model feedback systems — Every dynamic must have at least one feedback loop. No feedback = probably a mechanic, not a dynamic
  2. Trace to aesthetics — Every dynamic must list which AES specs it serves
  3. Invariants are non-negotiable — Hard constraints. Violations mean the code is wrong
  4. Name degenerate dynamics — Anticipate what can go wrong (the Monopoly problem)
  5. Be concrete — Use patterns with triggers, sequences, and durations
  6. No code — Dynamic specs describe behavior, not implementation

Quality Gate: Dynamic Readiness

npx mda gate dynamic
