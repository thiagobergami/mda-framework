Mechanic Spec Schema

Mechanic specs define the actions, behaviors, and control mechanisms afforded to the player. Together with game content, they support the dynamics. They answer: "What can the player DO and what RULES govern the system?"

File pattern: specs/mechanics/{feature}.mec.md
ID format: MEC-{NNN}
Scaffold: npx mda new mechanic "System Name"

Never "in vacuo"

The paper's core rule: components cannot be evaluated in isolation. Every mechanic MUST trace to at least one dynamic. A mechanic without a dynamic has no purpose.

Template

Frontmatter

---
id: MEC-{NNN}
name: {Descriptive name}
traces_to_dynamics: [DYN-NNN]
---

Required Sections

Purpose

1-2 sentences explaining WHY this mechanic exists — what dynamic it supports and what aesthetic it ultimately serves.

Player Affordances

The verbs — what the player can DO.

- Carry: Pick up and carry the lantern — Input: proximity + interact key
- Aim: Point the lantern beam — Input: mouse/stick direction
- Merge: Combine lantern light with nearby player — Input: automatic within 5 studs

Game Content

The material that mechanics operate on. Each entry references an asset spec by ID.

- Lantern model: Handheld light source — Asset: AST-007 — Status: placeholder
- Creature variants: 8 species — Asset: AST-008 — Status: concept

Rules

Constraints and formulas governing mechanics. Tunable parameters are marked [TUNABLE].

### Rule 1: Light Cone
- Condition: Player is carrying lantern
- Effect: Cone of light extends from lantern in aim direction
- Formula: intensity = basePower * (1 - distance/maxRange)
- Parameters:
  - basePower: 1.0 [TUNABLE]
  - maxRange: 20 studs [TUNABLE]
  - coneAngle: 45 degrees [TUNABLE]

Behavioral Contract

The specification that code must satisfy.

### Inputs
- playerPosition: Vector3 from character
- aimDirection: Vector3 from mouse/stick

### Outputs
- lightCone: Cone geometry for intersection tests
- illuminatedCreatures: list of creatures in the cone

### Guarantees
- Light cone updates every frame
- Creatures are never missed (no frame-skip holes)

Acceptance Criteria

Concrete, testable pass/fail conditions.

  • Light cone renders at 60fps on target hardware
  • Creature at maxRange - 1 stud is detected
  • Creature at maxRange + 1 stud is NOT detected
  • Merged lanterns increase effective range by mergeBonus factor

Integration Points

- Fires: CREATURE_LIT, CREATURE_REVEAL, LANTERN_MERGE
- Listens: PlayerAdded, CharacterAdded
- Reads: creature positions, player positions
- Writes: creature visibility state, journal entries

Engine Bindings

References to binding specs for each target engine.

Rules

  1. Never in vacuo — Every mechanic MUST have traces_to_dynamics
  2. Player affordances first — Start with what the player CAN DO
  3. Behavior over data — Describe what the system DOES, not just data structures
  4. Mark tunable parameters — Any numeric value that might need adjustment
  5. Acceptance criteria are tests — Specific enough to write automated tests from
  6. Integration points prevent surprises — Document all side effects
