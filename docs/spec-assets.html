Asset Spec Schema

Asset specs define requirements for game content — 3D models, animations, sounds, music, textures, particles, and UI elements. They bridge the gap between the designer's aesthetic intent and the artist's creative output.

File pattern: specs/assets/{name}.asset.md
ID format: AST-{NNN}
Scaffold: npx mda new asset "Asset Name"
Catalog: specs/assets/catalog.md

Template

Frontmatter

---
id: AST-{NNN}
name: {Descriptive name}
type: model | animation | sound | music | texture | particle | ui
traces_to_mechanics: [MEC-NNN]
traces_to_aesthetics: [AES-NNN]
status: concept | placeholder | draft | final
---

Required Sections

Purpose

Why this asset exists. Use aesthetic vocabulary: "this sound delivers Sensation during discovery reactions" not "this is a fun sound effect."

Emotional Intent

The artistic brief. What should the player FEEL?

- Primary emotion: warmth and safety
- Energy level: Low
- Tone: cozy, nurturing
- Reference: the feeling of holding a small sleeping animal

Technical Requirements

Hard limits per asset type: poly count, texture resolution, rig type, duration, format, spatial settings, etc.

Variants

Multiple states or variations with their trigger conditions.

Engine Integration

Container location, instance type, tags, attributes, and binding reference.

Placeholder Protocol

What to use until the final asset is ready. Placeholders must be behaviorally equivalent — the game must run correctly with them.

- Placeholder: Simple sphere with color-coded glow
- Behavioral equivalence: Same hitbox, same tag, same attributes
- Swap protocol: Replace MeshPart, keep tags and attributes

Status Lifecycle

concept  →  placeholder  →  draft  →  final
   |             |             |          |
   |             |             |          Artist-approved, perf-validated
   |             |             Created but not yet approved/optimized
   |             Temporary stand-in, behaviorally equivalent
   Requirement defined, nothing created yet

AI can implement against placeholder status and above. At concept status, write code structure but add a -- PLACEHOLDER NEEDED: AST-XXX comment.

Asset Naming Convention

{Category}_{AssetName}_{Variant}

Examples:
  Character_Firefly_Idle
  Sound_Reveal_Tier1
  Particle_LanternGlow_Merged

Rules

  1. Trace to mechanic AND aesthetic — Assets without a mechanic are unused; assets without an aesthetic have no design purpose
  2. Emotional intent is required — Artists and AI need to know the target feeling
  3. Technical requirements are hard limits — Prevents performance disasters
  4. Placeholders are behavioral equivalents — Same hitbox, same tags, same duration
  5. Status tracking is mandatory — The catalog reflects current state
  6. Naming follows the catalog convention — AI can find assets programmatically
