Engine Binding Schema

Binding specs define the engine-specific mapping between MDA behavioral specs and a target game engine. They answer: "How does this mechanic or asset translate to engine X?"

File pattern: specs/bindings/{name}.bind.md
ID format: BIND-{NNN}
Scaffold: npx mda new binding "System Roblox Binding"

Template

Frontmatter

---
id: BIND-{NNN}
name: {Descriptive name}
engine: roblox | unity | unreal
binds_to: [MEC-NNN, AST-NNN]
language: luau | csharp | cpp | blueprint
---

Required Sections

Bound Specs

Which behavioral specs this binding covers and what aspect it maps.

Scene Structure

How the behavioral model maps to the engine's scene graph. Includes an instance hierarchy diagram.

Tags & Markers

How game objects are identified and queried at runtime.

ConceptEngine Mapping
creature tagCollectionService tag "ForestCreature"
reveal stateInstance:GetAttribute("RevealState")

API Surface

Engine-specific services, APIs, and events used.

Data Representation

How game state maps to engine data types and locations.

Network Model

Authority (server/client/shared), replication strategy, input handling.

Platform Considerations

Performance budgets, mobile constraints, etc.

Rules

  1. One binding per engine per spec set — Don't combine Roblox and Unity in one file
  2. Reference behavioral specs — binds_to must list actual MEC/AST IDs
  3. No behavioral logic — Bindings describe HOW, not WHAT. The WHAT is in MEC/DYN specs
  4. Engine idioms — Use the target engine's standard patterns
  5. Keep in sync — When a MEC spec changes, review its bindings
  6. Completeness — Every behavioral contract aspect should have a mapping

For a quick reference of cross-engine mappings, see Multi-Engine Support.
