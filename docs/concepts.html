Core Concepts

The MDA framework replaces vague design language with a formal, layered approach to game development. Understanding these concepts is essential for writing effective specs.

What is MDA?

MDA stands for Mechanics, Dynamics, Aesthetics — a formal framework for understanding games created by Robin Hunicke, Marc LeBlanc, and Robert Zubek. The core insight is that games are artifacts, not media. The content of a game is its behavior, not the media that streams out of it towards the player.

The Three Layers

Layer Question Description
Mechanics HOW does it work? The actions, behaviors, and control mechanisms afforded to the player. Together with game content (levels, assets, etc.), they are the building blocks.
Dynamics WHAT emerges? The runtime behavior of mechanics acting on player input and each other over time. Modeled as feedback systems.
Aesthetics WHY does it exist? The desirable emotional responses evoked in the player. The target player experience.

The Dual Perspective

MDA has two directions of reading. Both are essential:

Designer Perspective (M → D → A) — Use when IMPLEMENTING

Mechanics  --give rise to-->  Dynamics  --which produce-->  Aesthetics
  1. Read the mechanic spec — implement the rules, actions, and control mechanisms
  2. Validate against the dynamic spec — does the intended behavior emerge?
  3. Validate against the aesthetic spec — does it produce the target experience?

Player Perspective (A → D → M) — Use when DEBUGGING

Aesthetics  --set the tone for-->  Dynamics  --which expose-->  Mechanics
  1. Start from the aesthetic spec — what experience is broken or missing?
  2. Trace to dynamics — which feedback system is misfiring?
  3. Trace to mechanics — which rule, parameter, or action needs adjustment?
Key Insight

Designers think M→D→A (bottom-up: "I'll add this rule and it will create this behavior"). Players experience A→D→M (top-down: "this feels wrong — what's causing it?"). Both directions are necessary for effective game development.

The 8 Aesthetic Categories

The MDA paper replaces vague words like "fun" with a directed vocabulary of 8 aesthetic categories. Every spec in this framework uses these categories.

# Aesthetic Frame Examples
1 Sensation Game as sense-pleasure Rez, Tetris Effect, Flower
2 Fantasy Game as make-believe Skyrim, The Sims, Animal Crossing
3 Narrative Game as drama Last of Us, Final Fantasy, Celeste
4 Challenge Game as obstacle course Dark Souls, Quake, Super Meat Boy
5 Fellowship Game as social framework Among Us, Charades, MMOs
6 Discovery Game as uncharted territory Outer Wilds, Zelda: BotW, Minecraft
7 Expression Game as self-discovery Minecraft Creative, LittleBigPlanet
8 Submission Game as pastime Stardew Valley, Cookie Clicker

Usage Rules

  • Every aesthetic spec lists its primary aesthetic (the dominant experience goal)
  • Additional aesthetics are listed as secondary (supporting, but not the focus)
  • When aesthetics conflict (e.g., Challenge vs Submission), the primary wins
  • Every design trade-off is resolved by asking: "which aesthetic is primary?"
Banned Vocabulary

Never use vague words like "fun", "gameplay", "engaging", or "cool" in specs. Instead, name the specific aesthetic category. "The player has fun" becomes "the player experiences Discovery through escalating reveals."

Feedback Systems

Dynamics are modeled as feedback systems — loops where outputs feed back as inputs.

Positive Feedback Loops

Advantage amplifies itself (rich get richer). Creates decisive outcomes but can destroy tension if unchecked. Example: Monopoly — owning more property generates more rent, which buys more property.

Negative Feedback Loops

Advantage is dampened (rubber-banding). Keeps players competitive but can feel unfair if too aggressive. Example: Mario Kart — leading players get weaker items, trailing players get powerful ones.

The Monopoly Problem

The MDA paper's key example: Monopoly's unchecked positive feedback loop (property → rent → more property) creates a "runaway leader" dynamic that destroys both Challenge and Fellowship aesthetics. Always ensure positive feedback loops have counteracting negative loops or caps.

Key Vocabulary

Dynamic Terms

TermDefinition
Emergent behaviorA dynamic that arises from mechanic interactions but is not explicitly coded. Example: "camping" in shooters.
Intended dynamicA behavior the designer explicitly targets. Specified in dynamic specs.
Degenerate dynamicA behavior that undermines aesthetic goals. Example: Monopoly's runaway leader problem.
InvariantA condition that must ALWAYS hold during a dynamic. Used as validation constraints.

Mechanic Terms

TermDefinition
MechanicAn action, behavior, or control mechanism afforded to the player within a game context.
Game contentLevels, assets, spawn points, items — the material that mechanics operate on.
Player affordanceWhat the player can DO — the verbs of the game (run, jump, shoot, build, trade).
RuleA constraint or formula that governs how mechanics interact.
ParameterA tunable numeric value within a rule.

Spec ID Conventions

Each spec gets a unique ID used for traceability:

LayerPrefixExample
Game ConceptGAME-GAME-001
AestheticsAES-AES-001
DynamicsDYN-DYN-001
MechanicsMEC-MEC-001
TuningTUN-TUN-001
AssetsAST-AST-001
BindingsBIND-BIND-001

IDs are sequential within their layer and are never reused, even if the spec is deleted. Use npx mda new to auto-assign IDs.

Traceability

Every spec layer links to adjacent layers through frontmatter fields like traces_to_dynamics or traces_to_aesthetics. This creates a bidirectional graph that enables both design directions: 

GAME-001 {Game Concept}
  |
  +--> AES-* {Aesthetic Specs}
  |      |
  |      +--> DYN-* {Dynamic Specs}
  |             |
  |             +--> MEC-* {Mechanic Specs}
  |             |      +--> TUN-* {Tuning Specs}
  |             |      +--> AST-* {Asset Specs}
  |             |
  |             +--> ... (more mechanics)
  |
  +--> ... (more aesthetics)

The specs/traceability.md file provides a master matrix of all links. The validation rule trace-resolution ensures all references resolve to existing spec IDs.

No Mechanics "In Vacuo"

The paper's core rule: components cannot be evaluated "in vacuo" — isolated from their effect on dynamics and player experience. Every mechanic MUST trace to at least one dynamic. The no-vacuo validation rule enforces this.
