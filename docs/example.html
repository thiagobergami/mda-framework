Walkthrough: Lantern Woods

A complete example of the framework in action — from rough game idea to validated implementation. The game is fictional, but every step reflects the real process.

The Game

Lantern Woods — a co-op exploration game where 2-4 players carry lanterns through a dark forest, revealing hidden creatures that react to light. Each creature reacts differently to light. The goal is to fill a "Field Journal" by discovering all creature types. Calm, no combat, ages 10+. Target platform: Roblox.

Phase 1: Concept Refinement

The designer describes a rough idea. The framework enters a refinement loop that pressure-tests the idea against the 6 Concept Readiness checks.

The conversation

Designer:  I want to make a co-op game where 2-4 players explore a dark forest
           carrying lanterns. The lanterns reveal hidden creatures in the
           environment. Each creature reacts differently to light. The goal is
           to fill a "Field Journal" by discovering all creature types.

Claude:    I've drafted the concept. Let me run the readiness gate:

           npx mda gate concept

           [PASS] Vision clarity
           [FAIL] Aesthetic commitment
                  Problem: Only 1 aesthetic marked Absent. The gate requires
                  at least 2 to ensure focus.
           [PASS] Core loop coherence
           [WARN] Boundary definition
                  Only 2 explicit boundaries. Gate requires at least 3.
           [PASS] Feature traceability
           [PASS] Scope realism

           RESULT: BLOCKED (1 fail, 1 warn)

Claude proposes fixes: mark Expression and Submission as Absent (players observe, not create; sessions have a clear Journal goal). Add a third boundary about competition. The designer approves and adds a fourth boundary about crafting.

npx mda gate concept

[PASS] Vision clarity
[PASS] Aesthetic commitment  (3 active, 3 absent)
[PASS] Core loop coherence
[PASS] Boundary definition   (4 boundaries with rationale)
[PASS] Feature traceability
[PASS] Scope realism

RESULT: PASSED (6/6)
What the gate caught

Without the gate, the concept would have tried to serve 5 aesthetics. That's too many — downstream specs would have conflicting goals. By forcing Absent aesthetics, the concept now has clear priorities that cascade into focused specs.

Phase 2: Gated Spec Authoring

Each layer is authored, then its gate runs before the next layer starts.

Aesthetics → Aesthetic Gate

Checking AES-003 (Creature Wonder):
  [PASS] 3 observable proxies, all measurable
  [FAIL] Anti-pattern "reveal fatigue" has no detection signal
         How do you DETECT "players stop being impressed by reveals"?

RESULT: BLOCKED

Claude proposes a measurable detection signal: "Average player pause duration at reveal drops below 0.5s for 3 consecutive reveals." This is trackable — the MDA logger records how long players stop moving after a creature is revealed. Designer approves, gate passes.

Dynamics → Dynamic Gate

Checking DYN-002 (Light Reveal Cycle):
  [WARN] Journal Momentum is a positive loop with no cap identified
         What prevents "journal completionism" from dominating?

RESULT: PASSED with warnings

Designer adds a cap: "Journal shows max 2 undiscovered silhouettes at a time, preventing overwhelm. Momentum comes from quality of each discovery, not quantity pressure."

Mechanics → Implementation Gate

Checking MEC-004 (Lantern System):    All checks pass
Checking MEC-005 (Creature Behavior):  All checks pass
Checking MEC-006 (Field Journal):      All checks pass
Checking bindings (engine: roblox):    All 3 bindings present
Checking tuning:                       All parameters documented

RESULT: PASSED — Ready for implementation

Phase 3: Implement-Debug Loop

Implementation and debugging form a continuous cycle, not a waterfall.

Iteration 1: Initial Implementation

Claude reads the spec chain (MEC-004 → DYN-002 → AES-002), reads the binding spec, writes Luau code with logger calls, syncs to Roblox Studio via Rojo.

Designer playtests: "The reveals don't feel rewarding. Players find creatures but just keep walking."

Analyzing the logs

[A:AES-002] [PASS] proxy=reveal_rate       value=52   target=50
[A:AES-003] [FAIL] proxy=reveal_pause      value=0.3  target=1.0  OUT_OF_RANGE

[D:DYN-002] INV-2: FAIL — 4/12 reveals had no partial reaction

Tracing cid=9:
[01:42.100] [M:MEC-004] CREATURE_LIT  light=0.12
[01:42.300] [M:MEC-004] CREATURE_LIT  light=0.28
[01:42.400] [M:MEC-004] CREATURE_LIT  light=0.74  ← jumped!
[01:42.401] [M:MEC-005] CREATURE_REVEAL hadPeek=false
[01:42.401] [D:DYN-002] FAIL INVARIANT inv=INV-2

Root cause: Light jumped from 0.28 to 0.74 in one frame, skipping the peek phase. peekThreshold (0.3) and revealThreshold (0.7) are too close for fast-moving players.

Fix: Lower peekThreshold from 0.3 to 0.15 (tuning) + add peekMinDuration = 1.0s to MEC-005 (mechanic amendment).

Iteration 2: Retest

[A:AES-003] [PASS] proxy=reveal_pause  value=1.2  target=1.0
[D:DYN-002] all invariants PASS
[A:AES-002] [FAIL] proxy=coop_clustering  value=0.15  target=0.70

Creatures work now, but co-op is broken — players ignore each other.

Iteration 3: A Vision Change

The A→D→M trace reveals this isn't a tuning or bug issue — there's no mechanical reason to cooperate. Every creature is soloable.

Vision Change Proposal

Trigger: coop_clustering at 0.15 (target 0.70)
Root cause: No creature requires 2+ lanterns. Solo always works.
Proposed change: Add rare creatures requiring merged lantern light + speed bonus for merged reveals on normal creatures.
Cascade: DYN-003 → MEC-005 → TUN-003 → AST-007 (4 specs modified)

The designer decides: "Do both — rare creatures require merge, AND merge makes normal reveals faster. Solo players can still complete 80% of the Journal."

Claude modifies specs top-down through gates, re-implements MEC-005, updates tuning and traceability.

Iteration 4: Validated

[A:AES-002] [PASS] proxy=coop_clustering  value=0.71  target=0.70
[A:AES-002] [PASS] proxy=reveal_rate      value=48
[A:AES-003] [PASS] proxy=reveal_pause     value=1.4
[D:DYN-002] all invariants PASS
[D:DYN-003] all invariants PASS

ALL PROXIES IN_RANGE. ALL INVARIANTS PASS.

Key Insight

The human makes design decisions (what the game should feel like). The AI makes analytical decisions (which proxy is failing, what the root cause is). The tooling makes structural decisions (are gates passing, are traces valid). No single actor has full authority. The framework makes these handoff points explicit.
